## 왜 하는가?

### 배경

지금은 한국어로 생각을 정리해 블로그에 쓰고 있지만, 별도로 영어·일본어·중국어를 공부하려면 같은 내용을 다시 교재나 번역 도구로 옮겨야 한다. 이 과정은 글쓰기와 언어 공부를 서로 다른 일로 만들기 때문에 꾸준히 이어가기 어렵다.

한국어 원문을 AI가 외국어로 바꾸기만 하면 게시 속도는 빨라지지만, 내가 어떤 표현이 자연스러운지 이해하거나 직접 고치는 학습은 남지 않는다. 번역에는 문맥과 말투에 따라 여러 정답이 있고 AI가 원문의 뉘앙스를 지나치게 매끈하게 만들거나 다른 의미로 옮길 수도 있다.

따라서 한국어 글을 기준으로 번역본, 표현 대안, 한국어 뉘앙스 설명, 핵심 어휘를 한곳에서 비교하고 수정하는 흐름이 필요하다. 최종 번역본은 AI 결과가 아니라 내가 검토하고 승인한 글이어야 한다.

### 목표

한국어 원문 한 편을 중심으로 영어·일본어·중국어 번역본을 만들고, 각 번역의 표현과 뉘앙스를 이해해 직접 고친 뒤 독자에게 원하는 언어로 제공한다.

### 단위 액션별 기대 효과

- 한국어 원문에서 언어별 AI 초안을 생성한다 → 빈 화면에서 번역을 시작하는 부담을 줄인다.
- 원문과 번역문을 나란히 비교하고 직접 수정한다 → 내 문체와 의도를 유지하면서 능동적으로 학습한다.
- 어색하거나 선택지가 있는 표현에 한국어 설명과 대안을 붙인다 → 단순한 정오 판단을 넘어 표현별 뉘앙스를 이해한다.
- 글에 실제로 쓰인 핵심 어휘와 표현을 정리한다 → 내 관심사와 문맥에 연결된 단어를 반복해서 익힌다.
- 방문자가 언어를 전환해 같은 글을 읽는다 → 하나의 URL과 글 정체성을 유지하면서 다국어 독자를 지원한다.

### 성공 기준

- 한국어 원문에서 영어·일본어·중국어 초안을 각각 생성할 수 있다.
- 다른 언어의 기존 수정본을 덮어쓰지 않고 언어별로 수정·저장·승인할 수 있다.
- 각 번역에서 AI가 주의가 필요한 표현, 추천 대안, 차이를 설명하는 한국어 메모를 제공한다.
- 글에 쓰인 핵심 단어·숙어를 문장 속 용례와 한국어 뜻으로 확인할 수 있다.
- 독자는 한 글 안에서 한국어·영어·일본어·중국어를 선택할 수 있고 선택한 언어가 페이지 이동 후에도 유지된다.
- 실제 한국어 글 한 편을 세 언어로 번역·수정·발행하며, 사용자가 표현을 이해하지 못해 외부 번역기로 다시 옮겨야 하는 핵심 구간이 없는지 확인한다.

## 핵심 경험

### 관리자 흐름

1. 별도 번역 페이지로 이동하지 않고 기존 블로그 화면에서 한국어·영어·일본어·중국어 중 전체 표시 언어를 선택한다.
2. 영어를 선택하면 피드와 글 상세의 제목·본문이 영어 번역본으로 바뀐다.
3. 현재 읽고 있는 글 옆의 관리자 전용 표현 패널에서 AI가 다음 학습 정보를 구조화해 제안한다.
   - 의미가 달라질 위험이 있거나 직역하기 어려운 원문 구간
   - 현재 번역을 선택한 이유
   - 사용할 수도 있는 대체 표현
   - 대체 표현마다 달라지는 격식, 감정, 강도, 자연스러움에 대한 한국어 설명
   - 글에서 배울 핵심 단어와 표현
4. 필요할 때 한국어 원문으로 전환해 의미를 비교하고 영어 번역본을 직접 수정한다.
5. 학습 메모와 어휘 항목도 수정·삭제·추가한다.
6. 언어별 번역본을 `초안` 또는 `공개` 상태로 저장한다.
7. 한국어 원문이 바뀌면 기존 번역에 `원문 변경됨` 표시를 보여 주고, 사용자가 선택한 경우에만 다시 번역한다.

### 방문자 흐름

1. 글 상단에서 한국어·English·日本語·简体中文 중 원하는 언어를 선택한다.
2. 같은 글과 URL 맥락을 유지한 채 선택한 언어의 제목과 본문을 읽는다.
3. `표현 공부`를 열어 현재 언어의 핵심 단어·숙어, 읽는 법, 한국어 뜻, 글 속 예문, 뉘앙스 메모를 본다.
4. 아직 공개되지 않은 언어는 선택지에서 비활성화하거나 한국어 원문으로 안전하게 돌아간다.

## 제품 원칙

- 한국어 원문이 글 의미의 기준이다.
- AI 번역은 항상 초안이며 자동 공개하지 않는다.
- 사용자가 수정한 번역을 재생성으로 조용히 덮어쓰지 않는다.
- “더 좋은 표현”을 하나의 정답처럼 단정하지 않고 맥락별 차이를 한국어로 설명한다.
- 어휘 학습은 사전 전체를 복제하지 않고 이 글에서 배울 가치가 있는 표현에 집중한다.
- 번역문도 한국어 원문과 같은 Markdown 표현과 문단 구조를 가능한 한 보존한다.
- 공개 독서 화면은 학습 기능 때문에 산만해지지 않도록 본문과 `표현 공부`를 분리한다.
- UI/UX는 문서로 완전히 정한 뒤 한 번에 구현하지 않고, 실제 글로 동작하는 작은 화면을 직접 써 보며 결정한다.

## UI/UX 검증 방식

첫 기준 글은 기존 공개 글 `글쓰기`(`writing`)로 하되, UI는 한 글만 분리한 작업실이 아니라 기존 블로그 전체에 적용한다. `글쓰기`는 짧은 행과 문단, 추상적인 표현, 비유와 감정의 강도가 함께 있어 번역과 뉘앙스 설명을 시험하기 좋다.

첫 테스트는 완성된 기능의 최종 검수가 아니라 다음 핵심 동선을 작은 단위로 직접 써 보는 방식으로 진행한다.

1. 기존 블로그 화면 우측 상단에서 전체 언어를 English로 바꾼다.
2. 피드의 모든 글이 영어 제목·본문으로 바뀌는지 읽는다.
3. 스크롤해 현재 글이 바뀔 때 우측 표현 패널도 같은 글의 표현으로 바뀌는지 확인한다.
4. 특정 번역 표현을 왜 썼는지 한국어 설명과 대안을 확인한다.
5. 한국어로 다시 전환했을 때 같은 블로그 위치와 탐색 맥락을 유지하는지 확인한다.

각 단계는 데스크톱과 모바일에서 실제로 실행 가능한 화면으로 만든다. 사용자가 멈추거나 원문과 번역의 대응 관계를 잃거나 저장 여부를 불안해한 지점을 기록하고, 그 결과를 다음 UI와 데이터 구조에 반영한다. 정적인 와이어프레임의 모양만 승인받거나 백엔드를 모두 만든 뒤 UI를 처음 확인하는 방식은 피한다.

### 현재 프로토타입 상태

- 별도 번역 페이지를 제거하고 관리자 로그인 상태의 기존 홈 피드와 글 상세에 국기와 큰 언어명을 함께 쓰는 전역 선택기를 넣었다. 현재 한국어·English는 동작하고 日本語·简体中文은 준비 중 상태로 보인다.
- English를 선택하면 현재 공개 글 10편의 제목·본문이 UI 검증용 영어 번역으로 바뀐다.
- 평상시에는 표현 설명을 숨기고 English 글의 `Edit`을 누른 경우에만 선택한 글의 표현 설명·대안을 연다.
- 데스크톱에서는 기존 블로그 우측 열에 붙고, 좁은 화면에서는 불투명한 하단 패널로 열린다.
- 이 단계의 영어 번역과 표현 정보는 코드에 둔 UI fixture다. 실제 OpenAI 호출, DB 저장·수정·공개, 일본어·중국어는 아직 구현하지 않았다.
- 영어 fixture를 원문 글 수정 API로 잘못 저장하지 않도록 English 모드에서는 기존 한국어 수정 버튼을 숨긴다.

## 기술 접근 초안

### 데이터 구조

기존 `posts`는 글의 공통 정체성과 한국어 원문을 유지한다. 별도 하위 구조로 번역과 학습 항목을 연결하는 방향을 우선 검토한다.

- `post_translations`
  - 글 slug와 언어 코드
  - 번역 제목·부제·Markdown 본문
  - `draft/public` 상태
  - 번역 기준이 된 원문 버전 또는 해시
  - AI 생성 시각, 사용자 수정 시각, 공개 시각
  - 사용한 모델과 프롬프트 버전
- `translation_notes`
  - 대상 언어와 원문/번역 구간
  - 현재 표현, 대안 표현, 한국어 설명
  - 유형: 의미, 말투, 격식, 문화적 맥락, 자연스러움 등
  - 표시 순서와 사용자 편집 여부
- `vocabulary_items`
  - 대상 언어, 표제어, 글에 나온 형태
  - 읽는 법 또는 발음 보조 정보
  - 한국어 뜻, 글 속 예문, 사용 메모
  - 난이도 또는 중요도, 표시 순서

구체적인 테이블 정규화 수준은 첫 화면 흐름과 AI 응답 구조를 검증한 뒤 확정한다. 번역·메모·어휘를 처음부터 하나의 큰 JSON 필드에만 저장하면 부분 수정과 검색이 어려워질 수 있고, 지나치게 잘게 나누면 초기 구현이 복잡해질 수 있다.

### OpenAI 연동

- OpenAI 호출은 브라우저가 아니라 서버 API에서만 수행하고 API 키를 클라이언트에 노출하지 않는다.
- 관리자 세션과 CSRF 검사를 통과한 요청만 번역을 생성할 수 있다.
- 모델 응답은 번역문, 뉘앙스 메모, 대안, 어휘를 구분한 구조화된 형식으로 받는다.
- 생성 요청에는 한국어 원문, 대상 언어, 글의 문체 지침, 기존 사용자 수정본 보존 여부를 명시한다.
- 긴 글은 Markdown 구조가 깨지지 않는 단위로 나누되 전체 문맥과 용어 일관성을 유지한다.
- 모델명은 배포 환경에서 사용 가능한 OpenAI API 모델 ID로 설정 가능하게 두고, 사용자가 요청한 ChatGPT 5.6에 대응하는 실제 API 모델 ID·가격·한도는 구현 직전에 확인한다.
- 실패, 시간 초과, 잘못된 구조 응답은 재시도 가능 상태로 남기며 기존 번역본은 보존한다.
- 프롬프트 인젝션 위험을 줄이기 위해 글 본문은 번역할 데이터로 취급하고 시스템 지침과 분리한다.

### URL과 언어 선택

첫 버전은 글 하나에 하나의 canonical slug를 유지하는 것을 원칙으로 한다. 언어 선택 표현은 다음 후보를 구현 전에 비교한다.

- 쿼리: `/posts/slug/?lang=en`
- 경로: `/en/posts/slug/`

검색 노출, 공유 URL, 언어 전환 유지, Astro 라우팅 복잡도를 비교해 결정한다. 공개 번역에는 올바른 `lang`, `hreflang`, canonical 메타데이터가 필요하다.

## 품질과 안전 기준

- 번역 생성 중이어도 기존 글 읽기와 수정 기능은 정상 동작한다.
- 같은 요청의 중복 제출로 번역 레코드가 중복 생성되지 않는다.
- 사용자 수정본을 덮어쓰는 작업은 명시적 경고와 확인 없이는 실행하지 않는다.
- 공개되지 않은 번역과 학습 메모는 방문자가 API나 직접 URL로 읽을 수 없다.
- 모든 언어의 Markdown은 기존 HTML/스크립트 제거 규칙을 그대로 적용한다.
- AI 오류를 사실처럼 보이지 않게 하고, 번역·설명·어휘를 모두 사용자가 수정할 수 있게 한다.
- 비용과 남용을 통제하기 위해 관리자별 생성 횟수 제한, 요청 크기 제한, 타임아웃을 둔다.
- API 키, 프롬프트 내부 정보, 비공개 글이 로그에 불필요하게 남지 않게 한다.

## 결정 사항

- [D1] 한국어를 기준 원문으로 유지한다.
- [D2] 영어·일본어·중국어 번역을 한 글에 연결한다.

- [D4] AI는 번역뿐 아니라 대체 표현과 뉘앙스 차이를 한국어로 설명한다.
- [D5] 방문자는 원하는 언어를 선택하고 해당 글의 어휘·표현을 학습할 수 있다.

- [D7] 중국어는 간체(`zh-CN`)로 시작하고 번체는 향후 필요할 때 확장한다.
